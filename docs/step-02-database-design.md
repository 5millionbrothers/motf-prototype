# 2단계 데이터베이스 설계

## 지금 단계의 의미

현재 단계는 16단계 중 2단계입니다.

목표는 moTF에서 다루는 정보를 Supabase 데이터베이스의 표로 정리하는 것입니다. 쉽게 말하면, 지금 코드 안에 직접 적혀 있는 숙소, 상품, 게시글, 댓글, 채팅 같은 정보를 앞으로는 데이터베이스에서 꺼내오게 만들기 위한 설계 단계입니다.

## 전체 구조

moTF는 처음에는 두 개의 화면으로 운영합니다.

- 이용자 사이트: 이용자가 숙소 예약, 공판장 주문, 커뮤니티, 채팅, 마이페이지를 사용합니다.
- 운영 포털: 사장님과 관리자가 같이 사용합니다.

하지만 권한은 처음부터 세 가지로 나눕니다.

- `user`: 이용자
- `partner`: 사장님
- `admin`: 우리 관리자

이렇게 해두면 지금은 운영 포털 하나를 같이 쓰더라도, 나중에 관리자 페이지를 따로 분리하기 쉽습니다.

## 핵심 원리

데이터베이스는 여러 개의 표로 나뉩니다.

예를 들어 숙소 예약은 혼자 존재하지 않습니다.

```text
회원이 있음
→ 숙소가 있음
→ 숙소 안에 객실이 있음
→ 회원이 객실을 예약함
→ 결제 기록이 생김
```

그래서 표도 아래처럼 연결됩니다.

```text
profiles
→ stays
→ rooms
→ reservations
→ payments
```

각 표는 `id`라는 고유번호를 갖고, 다른 표는 그 `id`를 저장해서 서로 연결합니다.

## 테이블 목록

### 1. profiles

회원 정보를 저장합니다.

저장하는 정보:

- 이름
- 연락처
- 학교/소속
- 권한: `user`, `partner`, `admin`
- 계정 상태

사용 화면:

- 이용자 마이페이지
- 운영 포털 회원 관리
- 권한 판단

### 2. partner_profiles

사장님 업체 정보를 저장합니다.

저장하는 정보:

- 업체명
- 대표자명
- 사업자번호
- 연락처
- 승인 상태

사용 화면:

- 운영 포털
- 사장님 계정 관리
- 관리자 승인 관리

### 3. stays

숙소 기본 정보를 저장합니다.

저장하는 정보:

- 숙소명
- 지역
- 주소
- 소개
- 최대 인원
- 기준 가격
- 지도 좌표
- 공개 여부
- 소유 사장님

사용 화면:

- 이용자 숙소 목록/상세
- 운영 포털 숙소 관리

### 4. rooms

숙소 안의 객실 또는 대관 상품을 저장합니다.

저장하는 정보:

- 객실명
- 수용 인원
- 가격
- 객실 설명
- 공개 여부

사용 화면:

- 숙소 상세 페이지
- 예약 페이지
- 운영 포털 객실 관리

### 5. markets

공판장 정보를 저장합니다.

저장하는 정보:

- 공판장명
- 지역
- 주소
- 소개
- 수령/배송 방식
- 지도 좌표
- 공개 여부
- 소유 사장님

사용 화면:

- 이용자 공판장 목록/상세
- 운영 포털 공판장 관리

### 6. products

공판장 상품 정보를 저장합니다.

저장하는 정보:

- 상품명
- 카테고리
- 가격
- 단위
- 원산지
- 재고
- 공개 여부

사용 화면:

- 공판장 상품 목록/상세
- 장바구니
- 운영 포털 상품 관리

### 7. reservations

숙소 예약 기록을 저장합니다.

저장하는 정보:

- 예약자
- 숙소
- 객실
- 날짜
- 인원
- 대표자 정보
- 체크인/체크아웃
- 상태
- 금액

사용 화면:

- 예약 완료 화면
- 마이페이지 이용 내역
- 운영 포털 예약 관리

### 8. orders / order_items

공판장 주문과 주문 상품 상세를 저장합니다.

`orders`는 주문 한 건 전체를 저장하고, `order_items`는 주문 안에 담긴 상품들을 저장합니다.

사용 화면:

- 공판장 결제
- 마이페이지 주문 내역
- 운영 포털 주문 관리

### 9. payments

결제 기록을 저장합니다.

저장하는 정보:

- 주문번호
- 결제키
- 결제 타입: 숙소, 공판장
- 결제 상태
- 금액
- Toss 응답값

사용 화면:

- 결제 성공/실패 확인
- 마이페이지
- 운영 포털 거래내역

### 10. boards / posts / comments / likes

커뮤니티 게시판을 저장합니다.

구조:

```text
boards: 나눔장터, 대결신청, 현장정보, 익명고백
posts: 게시글
comments: 댓글
likes: 공감
```

사용 화면:

- 커뮤니티
- 게시글 상세
- 운영 포털 게시글 관리

### 11. recreation_posts

추천 레크레이션을 저장합니다.

저장하는 정보:

- 제목
- 추천 인원
- 공간
- 분위기
- 진행 시간
- 설명
- 공감/댓글

사용 화면:

- 추천 레크레이션 목록
- 레크레이션 상세
- 운영 포털 콘텐츠 관리

### 12. chat_rooms / messages

채팅방과 메시지를 저장합니다.

저장하는 정보:

- 채팅 참여자
- 숙소/공판장 연결 정보
- 메시지
- 읽음 여부
- 첨부파일

사용 화면:

- 이용자 채팅
- 사장님 채팅
- 관리자 중개 채팅

### 13. reviews

리뷰를 저장합니다.

저장하는 정보:

- 작성자
- 대상 타입: 숙소, 공판장
- 평점
- 태그
- 후기 내용
- 공개 여부

사용 화면:

- 숙소/공판장 상세
- 마이페이지 리뷰
- 운영 포털 리뷰 관리

### 14. attachments

사진/영상 파일 정보를 저장합니다.

실제 파일은 Supabase Storage에 저장하고, 이 표에는 파일 주소와 연결 대상을 저장합니다.

사용 화면:

- 숙소 사진
- 상품 사진
- 채팅 첨부
- 게시글 첨부
- 레크레이션 첨부

### 15. disputes

문의, 신고, 분쟁을 저장합니다.

저장하는 정보:

- 접수자
- 분쟁 유형
- 연결된 예약/주문/게시글
- 상태
- 관리자 메모

사용 화면:

- 이용안내 문의하기
- 운영 포털 분쟁 관리

### 16. settlement_files

예결산 엑셀 파일 생성 기록을 저장합니다.

저장하는 정보:

- 생성한 사용자
- 연결된 예약/주문
- 파일 주소
- 양식 이름

사용 화면:

- 마이페이지 예결산 엑셀
- 운영 포털 정산 관리

## 2단계 결론

처음부터 모든 기능을 완벽하게 구현하지 않아도 됩니다.

하지만 표 구조는 처음부터 이 정도로 잡아두는 것이 좋습니다. 그래야 나중에 숙소, 공판장, 커뮤니티, 채팅, 결제 기능을 붙일 때 구조를 크게 갈아엎지 않아도 됩니다.

다음 작업은 이 설계를 Supabase SQL로 옮기는 것입니다.
