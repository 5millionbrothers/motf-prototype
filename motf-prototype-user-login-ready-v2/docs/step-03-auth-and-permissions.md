# 3단계 회원/권한 설계

## 지금 단계의 의미

현재 단계는 16단계 중 3단계입니다.

2단계에서 데이터베이스 표를 만들었다면, 3단계에서는 각 표를 누가 볼 수 있고 누가 수정할 수 있는지 정합니다. 쉽게 말하면 데이터베이스에 문지기를 세우는 단계입니다.

## moTF의 회원 역할

moTF는 화면은 처음에 두 개로 시작합니다.

- 이용자 사이트
- 운영 포털

하지만 로그인한 사람의 역할은 세 가지입니다.

| 역할 | 코드값 | 설명 |
| --- | --- | --- |
| 이용자 | `user` | 여행자, 학생회, MT 준비팀 |
| 사장님 | `partner` | 숙소 사장님, 공판장 사장님 |
| 관리자 | `admin` | 우리 moTF 운영자 |

## 기본 원칙

### 이용자

이용자는 공개된 정보와 자기 기록만 볼 수 있습니다.

볼 수 있는 것:

- 공개된 숙소/객실
- 공개된 공판장/상품
- 공개 커뮤니티 게시글
- 자기 예약
- 자기 주문
- 자기 채팅
- 자기 리뷰

할 수 있는 것:

- 예약 생성
- 주문 생성
- 커뮤니티 글/댓글 작성
- 공감
- 리뷰 작성
- 문의/분쟁 접수

### 사장님

사장님은 자기 업체와 연결된 데이터만 볼 수 있습니다.

볼 수 있는 것:

- 자기 숙소
- 자기 객실
- 자기 공판장
- 자기 상품
- 자기 숙소 예약
- 자기 공판장 주문
- 자기 고객과의 채팅
- 자기 업체에 달린 리뷰

할 수 있는 것:

- 자기 숙소/객실 수정
- 자기 상품 수정
- 예약/주문 상태 확인 및 처리
- 채팅 응답

제한되는 것:

- 다른 사장님 숙소/상품/거래내역 조회 불가
- 전체 회원 조회 불가
- 전체 게시글 관리 불가

### 관리자

관리자는 전체 데이터를 볼 수 있고 운영상 필요한 수정을 할 수 있습니다.

볼 수 있는 것:

- 전체 회원
- 전체 사장님
- 전체 숙소/공판장
- 전체 예약/주문/결제
- 전체 게시글/댓글/리뷰
- 전체 채팅
- 전체 문의/분쟁

할 수 있는 것:

- 사장님 승인
- 숙소/상품 공개/숨김
- 게시글/댓글 숨김
- 분쟁 상태 변경
- 운영 메모 작성
- 정산 파일 관리

## Supabase RLS란?

RLS는 Row Level Security의 줄임말입니다.

쉽게 말하면 “표 전체를 여는 게 아니라, 행 하나하나마다 볼 수 있는지 검사하는 보안 규칙”입니다.

예를 들어 `reservations` 표에 예약이 1,000개 있어도 이용자 A는 자기 예약 3개만 볼 수 있어야 합니다.

```text
전체 예약 표
→ 로그인한 사람 ID 확인
→ user_id가 내 ID인 예약만 보여줌
```

사장님은 이런 식입니다.

```text
전체 예약 표
→ 예약된 숙소 확인
→ 그 숙소의 partner_id가 내 사장님 계정과 연결되어 있으면 보여줌
```

관리자는 전체를 볼 수 있습니다.

```text
내 role이 admin이면 전체 허용
```

## 이번 단계의 결과물

이번 단계에서는 아래 파일을 만듭니다.

- `supabase/rls.sql`

이 파일은 Supabase SQL Editor에서 실행할 수 있는 권한 정책 초안입니다.

## 주의점

아직 이 SQL을 바로 운영 DB에 실행하지 않는 것이 좋습니다.

권한 정책은 한 번 틀리면 화면에서 데이터가 안 보이거나, 반대로 보이면 안 되는 데이터가 보일 수 있습니다. 그래서 다음 단계에서 운영 포털 MVP와 연결하면서 한 번 더 조정하는 것이 안전합니다.
